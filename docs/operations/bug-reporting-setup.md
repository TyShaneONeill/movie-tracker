# Bug Reporting — Operations Setup Runbook

This document is the post-merge deploy checklist for the `feat/bug-reporting-system` work. Run these steps once the PR is merged to enable end-to-end bug reporting in production.

**Supabase project ref:** `wliblwulvsrfgqcnbzeh`

---

## 1. Create the Discord webhook

1. Open Discord → PocketStubs server → `#bugs` channel (create if missing)
2. Channel settings → **Integrations** → **Webhooks** → **New Webhook**
3. Name: `PocketStubs Bug Reports`
4. Click **Copy Webhook URL** — paste into step 2 below as `DISCORD_WEBHOOK_BUGS_URL`

---

## 2. Set Supabase secrets

Eight secrets total. Two are new from the 2026-04-25 Sentry SDK refactor (`SENTRY_DSN`, `SENTRY_PROJECT_ID`); the rest are from the original plan.

### Where to find each value

| Secret | Where to get it |
|---|---|
| `DISCORD_WEBHOOK_BUGS_URL` | From step 1 above |
| `SENTRY_DSN` | Sentry → Settings → Projects → CineTrak → **Client Keys (DSN)** → copy the DSN URL |
| `SENTRY_PROJECT_ID` | Numeric ID at the end of the DSN (e.g. `https://...@o123.ingest.sentry.io/4567890` → `4567890`). Also visible at Sentry → Settings → Projects → CineTrak |
| `SENTRY_PROJECT` | Project slug (lowercase, hyphenated — e.g. `pocketstubs-mobile`). Sentry → Settings → Projects → CineTrak → field labeled "Slug" |
| `SENTRY_ORG` | `pocketstubs-5w` (visible in any Sentry URL: `https://pocketstubs-5w.sentry.io/...`) |
| `SENTRY_AUTH_TOKEN` | If an org-scoped auth token already exists (e.g. for releases), reuse it. Otherwise create one at `https://sentry.io/settings/account/api/auth-tokens/` with scopes `project:read`, `project:write`, `event:read` |
| `SENTRY_WEBHOOK_SECRET` | Generated by `openssl rand -hex 32` below — must match the value pasted into Sentry's webhook config in step 4 |
| `ANTHROPIC_API_KEY` | `https://console.anthropic.com/settings/keys` → reuse the existing CineTrak key if present |

### Set them

```bash
# Discord
supabase secrets set DISCORD_WEBHOOK_BUGS_URL='<paste-from-step-1>' --project-ref wliblwulvsrfgqcnbzeh

# Sentry — SDK ingestion (NEW)
supabase secrets set SENTRY_DSN='<paste-from-sentry-client-keys>' --project-ref wliblwulvsrfgqcnbzeh
supabase secrets set SENTRY_PROJECT_ID='<numeric-id-from-DSN-tail>' --project-ref wliblwulvsrfgqcnbzeh

# Sentry — REST reads + comment POST
supabase secrets set SENTRY_AUTH_TOKEN='<sentry-auth-token>' --project-ref wliblwulvsrfgqcnbzeh
supabase secrets set SENTRY_ORG='pocketstubs-5w' --project-ref wliblwulvsrfgqcnbzeh
supabase secrets set SENTRY_PROJECT='<project-slug>' --project-ref wliblwulvsrfgqcnbzeh

# Sentry webhook — generate fresh, then paste into Sentry config in step 4
export SENTRY_WEBHOOK_SECRET=$(openssl rand -hex 32)
echo "SENTRY_WEBHOOK_SECRET=$SENTRY_WEBHOOK_SECRET"   # <-- copy this for step 4
supabase secrets set SENTRY_WEBHOOK_SECRET="$SENTRY_WEBHOOK_SECRET" --project-ref wliblwulvsrfgqcnbzeh

# Anthropic
supabase secrets set ANTHROPIC_API_KEY='<anthropic-key>' --project-ref wliblwulvsrfgqcnbzeh
```

### Verify

```bash
supabase secrets list --project-ref wliblwulvsrfgqcnbzeh \
  | grep -E "DISCORD_WEBHOOK_BUGS_URL|SENTRY_DSN|SENTRY_PROJECT_ID|SENTRY_AUTH_TOKEN|SENTRY_ORG|SENTRY_PROJECT|SENTRY_WEBHOOK_SECRET|ANTHROPIC_API_KEY"
```

Expected: 8 lines.

## 4. Configure the Sentry feedback webhook

1. Sentry → **Settings → Integrations → Webhooks → Create New**
2. **URL**: `https://wliblwulvsrfgqcnbzeh.supabase.co/functions/v1/analyze-bug-report`
3. **Events**: check ONLY `user_feedback.created` — do NOT subscribe to `error.created` or others (would fire LLM analysis on every error)
4. **Secret**: paste the `SENTRY_WEBHOOK_SECRET` value generated in step 2
5. Save

## 6. Manual device + browser verification

After ops setup is healthy, run the manual checklist from `docs/superpowers/plans/2026-04-24-bug-reporting-system.md` — Task 21. Highlights:

- iOS Settings entry → Report a Bug → modal opens with screenshot of previous screen → submit → Sentry/Discord receive it
- iOS Shake entry → forceful shake → "Report a bug?" confirm → Yes → full modal with screenshot
- iOS Shake cooldown — re-shake within 10s should NOT re-trigger; after 10s should
- Web — Settings entry works, shake is a no-op (by design)
- Rate limit — submit 6 within an hour → 6th shows inline rate-limit error
- Offline — submit fails inline, modal does NOT auto-dismiss

## Known limitations (document in PR)

- **Duplicate submissions on retry**: server has no idempotency key. If a user clicks Submit again after a network failure recovers, a duplicate report IS created. Acceptable for MVP volume; revisit when feedback volume warrants.
- **Pre-existing `react-native-view-shot` web bundle leak**: from `lib/share-service.ts` and `app/review/[id].tsx` (not introduced by this PR). The lib warns rather than crashes when its native module is missing, so web works today. Worth a separate cleanup PR.
